Summary

Add a second overload of ForMessagesFromAssembly (and its …Containing<TMarker> sibling) that runs a caller-supplied callback once per discovered IConsume<TMessage>. The callback can route a consumer to the queue lane, set concurrency / group / handler-id / circuit-breaker, or skip it — replacing today's behaviour where every scanned consumer silently registers as a Bus consumer at concurrency 1. The existing no-arg overload and all current callers stay byte-for-byte unchanged.

Problem Frame

Assembly scanning is the ergonomic on-ramp for registering many consumers at once, but it is currently a one-shape door. MessageRegistrationFactory.CreateScanned hardcodes IntentType.Bus and concurrency 1 for every discovered consumer. A consumer authored for the queue lane — point-to-point work distribution, higher concurrency — that happens to live in a scanned assembly registers as a broadcast Bus consumer with no signal to the developer. The only escape today is to abandon scanning for that consumer and hand-register it with explicit ForMessage<T>, which defeats the purpose of scanning an assembly that legitimately mixes both lanes.

The cost lands at runtime, not compile time: the consumer is wired, starts on the wrong lane, and the misconfiguration surfaces as "messages aren't being load-balanced the way I expected" rather than a build error. There is no per-consumer seam in the scan path to set intent, concurrency, group, handler-id, or a circuit-breaker override, even though the underlying registration record already carries every one of those fields.

Key Decisions

Imperative callback, not declarative attributes

Configuration is supplied as Action<ScannedConsumerContext, IScannedConsumerBuilder>, not as attributes on consumer classes. Attributes would couple consumer types to Headless-owned attribute types, which cuts directly against the framework's unopinionated / zero-lock-in identity, and they cannot configure consumers the caller does not own. The callback strictly subsumes the attribute approach: a caller who wants attribute-driven config can read their own attributes inside the callback via ctx.ConsumerType.GetCustomAttribute<…>(). Declarative attributes were considered and rejected on this basis.

Scanning stays name-inference-only — no MessageName on the scanned builder

The scanned builder deliberately omits a MessageName(string) method. The conceptual split is: scanning infers message identity; explicit ForMessage<T> asserts it. A per-consumer name override inside a scan callback is the moment the caller has stopped scanning and started hand-registering — and explicit registration already wins via precedence, so the escape hatch exists without adding surface. This also sidesteps the cross-type name-collision sweep interaction the issue flagged, since the callback cannot introduce a new message name during scanning.

Skip() is fully exclusionary — suppresses DI registration too

Today the scan path registers DI descriptors (consumerType and IConsume<T>) for every discovered consumer before building its registration. For Skip() to mean "excluded entirely," the new overload must invoke the callback first and, when a consumer is skipped, register neither its MessageRegistration nor its DI descriptors. A skipped consumer is not resolvable through the scan path at all.

No assembly-wide blanket Group default

Group stays inferred per consumer. A blanket group applied across distinct message types produces competing consumers (a shared group across different messages is a load-balancing footgun), so no assembly-level Group default is offered — even per-consumer, group is normally left untouched and documented as such.

Scan-vs-explicit precedence unchanged; callback configures the scanned registration only

An explicit ForMessage<T> registration for the same (consumer, message) pair still wins. The callback shapes the scanned registration; discovery's two-pass name resolution, idempotent merge, and dedup are untouched. Discovery (_DiscoverMessageRegistrations) remains the single registration path — only the values CreateScanned produces change from hardcoded Bus/1 to resolved intent/group/concurrency/handler-id/circuit-breaker.

Registration Pipeline

Where the callback sits in the existing scan-to-registry path. The callback is the only new step; everything downstream of it is unchanged.

Requirements

API surface
IDRequirement
R1.Add overload ForMessagesFromAssembly(Assembly assembly, Action<ScannedConsumerContext, IScannedConsumerBuilder> configure).
R2.Add overload ForMessagesFromAssemblyContaining<TMarker>(Action<ScannedConsumerContext, IScannedConsumerBuilder> configure), delegating to R1 with the marker's assembly.
R3.Introduce ScannedConsumerContext as a read-only record struct exposing ConsumerType and MessageType. It carries no mutating surface — it is the read-only "who is this consumer" half passed to the callback.
R4.Introduce IScannedConsumerBuilder ([PublicAPI]) exposing OnBus() (default), OnQueue(), Group(string), Concurrency(byte), HandlerId(string), WithCircuitBreaker(Action<ConsumerCircuitBreakerOptions>), and Skip(). It deliberately has no MessageName method.
Behaviour & defaults
IDRequirement
R5.The callback runs exactly once per discovered closed IConsume<TMessage> implementation in the scanned assembly.
R6.A consumer the callback does not touch registers as Bus / concurrency 1 — bit-for-bit identical to today's no-arg overload. This is the regression guard the no-arg path relies on.
R7.Skip() excludes the discovered consumer entirely: neither its MessageRegistration nor its DI descriptors are registered through the scan path. The callback must therefore run before DI descriptor registration.
R8.Validation matches the explicit consumer builder: Concurrency(0) is rejected, and empty / whitespace Group and HandlerId are rejected, using the same Argument.* guard semantics so the two builders fail identically.
R9.HandlerId is the caller's responsibility. If the callback produces a duplicate identity within the same (messageName, group, intent), the existing competing-identity guard throws at startup — fail-fast is the intended outcome, not a new check.
Discovery integration
IDRequirement
R10.Scan-vs-explicit precedence is unchanged: an explicit ForMessage<T> registration for the same (consumer, message) pair wins; the callback shapes only the scanned registration that fills gaps.
R11._DiscoverMessageRegistrations stays the single registration path. Only the resolved fields produced for a scanned consumer change (intent / group / concurrency / handler-id / circuit-breaker instead of hardcoded Bus/1). Two-pass name resolution, idempotent merge, and dedup are untouched.
R12.The existing no-arg ForMessagesFromAssembly(Assembly) overload and all current callers are unaffected by this change.

Acceptance Examples

AE1. Covers R4, R5. A scanned consumer whose callback calls OnQueue() registers on the queue lane and is started by the registrar on the queue lane (not Bus).

AE2. Covers R5, R6. A single scan call over an assembly mixing lanes: consumers the callback flips with OnQueue() land on the queue lane; consumers it leaves untouched land on Bus — both resolved correctly from one ForMessagesFromAssembly invocation.

AE3. Covers R7. A consumer whose callback calls Skip() is absent from the registry and not resolvable from DI through the scan path.

AE4. Covers R6. A consumer the callback never touches resolves to Bus / concurrency 1 — identical to the no-arg overload's output for the same assembly.

AE5. Covers R10. When the same (consumer, message) pair is registered explicitly via ForMessage<T> and appears in a configured scan, the explicit registration wins; the scanned configuration (e.g. an OnQueue() call) does not override it.

AE6. Covers R8. Concurrency(0) throws; empty / whitespace Group and HandlerId throw — with the same exception type and behaviour as the explicit builder.

Scope Boundaries

Outside this feature's identity

Declarative attribute-based configuration. Considered and rejected — couples consumer classes to Headless attributes (lock-in) and cannot configure consumers the caller does not own. The callback subsumes it via ctx.ConsumerType.GetCustomAttribute<…>().

Outside this feature's identity

Assembly-wide blanket Group default. A shared group across distinct message types creates competing consumers. No assembly-level group default is offered; group stays inferred per consumer.

Deferred — escape hatch already exists

MessageName override on the scanned builder. Excluded by decision. Custom message names route through explicit ForMessage<T>, which wins via precedence. Revisit only if a concrete need for per-scan renaming emerges.

Deferred

Predicate / filter overloads beyond the callback (e.g. a separate type-filter parameter). The callback already supports broad-scan-then-prune via Skip(); a dedicated filter overload is not needed for the stated use case.

Dependencies & Assumptions

  • Builds on #362 (the ForMessage registration API). The underlying MessageConsumerRegistration record already carries every field the callback resolves — no new storage shape is required.
  • OnQueue-on-explicit no-op is accepted, not warned. If a callback flips OnQueue() on a pair also registered explicitly, the scanned registration is dropped by precedence (R10) and the call silently has no effect. This is consistent with the documented precedence model; it is documented as expected behaviour, not surfaced as a startup warning.
  • ctx.MessageType.Namespace can be null for types in the global namespace. Callback authors using namespace-based routing must use the null-safe access pattern; this is called out in the doc/XML comments for the overload.
  • Callback invocation order is not guaranteed. Discovery order follows assembly.GetTypes(), which has no stable ordering contract. The callback must be order-independent; this is a documentation note, not an enforced constraint.

Outstanding Questions

None blocking planning

No open product decisions remain. The MessageName include/exclude fork is resolved (exclude). The one item deferred to planning is purely mechanical: the exact reshaping of the scan loop so the callback runs before DI descriptor registration (R7) — an implementation-ordering detail, not a product question.
