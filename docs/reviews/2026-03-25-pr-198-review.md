# Code Review Summary

The storage-vs-message identity split is internally coherent, but the rollout and migration story is not production-safe yet. The main blockers are deployment compatibility around the new required published `MessageId` column, a SQL Server repair path that is not rerunnable after partial upgrades, and two contract regressions around published `MessageId` length and in-memory exception-message storage.

## Key Findings

- P1: the new required published `MessageId` column turns this into a full writer cutover with no safe rolling-deploy or code-only rollback path
- P2: SQL Server only repairs the new `MessageId` column when the column is missing, so partial upgrades can strand NULLs and break monitoring reads
- P2: durable providers still cap published `MessageId` at 200 characters with no fail-fast validation, despite documenting it as an arbitrary logical string
- P2: `InMemoryDataStorage.StoreReceivedExceptionMessageAsync(...)` now throws on raw string content; the focused unit test already fails
- P3: `IdentityMessageDescriber` constant cleanup is unrelated to the messaging plan and should be split or explicitly justified
