## Summary

PR adds two new NuGet packages (`Headless.AuditLog.Abstractions`, `Headless.AuditLog.EntityFramework`), integrates them into `HeadlessDbContext` and `HeadlessIdentityDbContext` via a two-phase audit save pipeline, and adds a prerequisite `ICorrelationIdProvider` to `Headless.Core`.

## Findings by Severity

### P1 — Critical (must fix before merge)

#### 001: Audit pipeline is NOT atomic on the common SaveChanges path

**Files:** `src/Headless.Orm.EntityFramework/Contexts/HeadlessDbContext.cs:48-51`, `HeadlessIdentityDbContext.cs:67,78`

The two-phase audit pipeline is non-atomic for the most common use case: when no message emitters are present.

Current flow (no-emitter path):
```
1. _BaseSaveChanges()           ← entity changes committed to DB
2. _ResolveAndPersistAudit()    ← audit entries added then _BaseSaveChanges() called again
```

If the second `_BaseSaveChanges` fails (DB error, transient fault), entities are permanently persisted but audit entries are lost. The PR description states "Atomic: Audit entries commit in the same SaveChanges transaction as entity changes" — this is true only for the explicit-transaction paths (when emitters are registered or `CreateExecutionStrategy` wraps both phases within an open transaction).

**Fix:** For the no-emitter path, when `auditEntries.Count > 0`, wrap both save phases in an explicit transaction:
```csharp
if (auditEntries is { Count: > 0 })
{
    await using var tx = await Database.BeginTransactionAsync(cancellationToken);
    var result = await _BaseSaveChangesAsync(acceptAllChangesOnSuccess, cancellationToken);
    await _ResolveAndPersistAuditAsync(auditEntries, cancellationToken);
    await tx.CommitAsync(cancellationToken);
    return result;
}
```

### P2 — Important (fix soon, ideally this PR or immediate follow-up)

#### 002: String fields lack truncation — UserAgent and others can cause DB overflow

**File:** `src/Headless.AuditLog.EntityFramework/EfAuditLogStore.cs:64`, `EfAuditLog.cs:41`

`CorrelationId` is defensively truncated at 128 chars. `UserAgent` (max 512), `EntityId` (max 256), `EntityType` (max 512), `Action` (max 256), `UserId` (max 128) have no truncation. A browser UserAgent with many extensions can exceed 512 chars. Passing such a value would throw a `DbUpdateException` at save time.

**Fix:** Introduce a `Truncate(string? value, int maxLength)` helper in `EfAuditLogStore._AddEntries` and apply to all bounded columns.

#### 003: AuditSavePipelineHelper is public — leaks internal implementation detail

**File:** `src/Headless.Orm.EntityFramework/AuditSavePipelineHelper.cs:16`

`AuditSavePipelineHelper` is `public static` in a framework package. This exposes save-pipeline internals as a supported public API surface. Downstream consumers could take an unintended dependency on `CaptureAuditEntries`, `SaveAuditEntries`, etc.

**Fix:** Make `internal`. Add `InternalsVisibleTo("Headless.Identity.Storage.EntityFramework")` in `Headless.Orm.EntityFramework.csproj`.

#### 004: EfReadAuditLog orders by `Id DESC` — semantically incorrect on composite-keyed table

**File:** `src/Headless.AuditLog.EntityFramework/EfReadAuditLog.cs:40`

The table has composite PK `(CreatedAt, Id)` and all indexes include `CreatedAt`. `OrderByDescending(e => e.Id)` is index-hostile (requires a sort pass) and semantically wrong for partitioned configurations where `Id` is not globally monotonic.

**Fix:** Change to `.OrderByDescending(e => e.CreatedAt).ThenByDescending(e => e.Id)`.

#### 005: _ResolveAndPersistAudit duplicated verbatim in two contexts

**Files:** `HeadlessDbContext.cs:459-482`, `HeadlessIdentityDbContext.cs:460-486`

Both contexts have identical private `_ResolveAndPersistAuditAsync` and `_ResolveAndPersistAudit` methods. Since `HeadlessIdentityDbContext` does not inherit `HeadlessDbContext`, the duplication is structural but still a maintenance liability.

**Fix:** Move the logic to `AuditSavePipelineHelper` accepting a `_BaseSaveChanges` delegate.

### P3 — Nice-to-Have (track for future)

#### 006: IReadAuditLog.QueryAsync has no cursor/offset pagination

**File:** `src/Headless.AuditLog.Abstractions/IReadAuditLog.cs:35-47`

`limit`-only pagination silently truncates large result sets with no indication of remaining data. No mechanism for consumers to page through audit history.

**Track:** Add cursor-based pagination (`beforeId` or `afterCreatedAt`) in a future version. Document this as a v1 limitation.

#### 007: Missing index on (TenantId, Action, CreatedAt)

**File:** `src/Headless.AuditLog.EntityFramework/AuditLogEntryConfiguration.cs:68-87`

The primary compliance query pattern (show all PII access events for a tenant in a time range) has no supporting index. `IReadAuditLog.QueryAsync(action: "pii.revealed", tenantId: ...)` would scan at scale.

**Track:** Add `HasIndex(e => new { e.TenantId, e.Action, e.CreatedAt }).HasDatabaseName("ix_audit_log_tenant_action_time")`.

#### 008: IAuditLog.LogAsync CancellationToken is ignored in EF implementation

**File:** `src/Headless.AuditLog.EntityFramework/EfAuditLog.cs:19-55`

The method is synchronous (always returns `Task.CompletedTask`) and the `CancellationToken` is not forwarded to any operation. The async signature is forward-compatible for non-EF implementations but could mislead consumers.

**Track:** Add XML doc clarifying that cancellation is forwarded only when the EF implementation has async I/O to perform (none in current implementation, reserved for future).

## Architecture Observations

**Strengths:**
- Clean abstraction + provider split. `IAuditChangeCapture`, `IAuditLogStore`, `IReadAuditLog` are properly decoupled from EF.
- The `_deferredEntityIds` pattern for store-generated key resolution is well-designed. Thread safety is correct for the Scoped lifetime.
- `_PropertyCache` as a `static ConcurrentDictionary<PropertyInfo, AuditPropertyMetadata>` is appropriate — property metadata is immutable and shared safely across instances.
- `AuditIgnore` on `AuditLogEntry` itself correctly prevents recursive capture in `AuditByDefault` mode.
- The `PrepareForRetry` / execution strategy integration is thoughtfully handled.
- `SensitiveValueTransformer` exception fallback to `Redact` with a warning prevents a transformer bug from blocking entity saves.
- `FluentValidation` in `AuditLogOptions` follows the established pattern in this codebase (via `Headless.Hosting` transitive dep).
- Test coverage is thorough: all key behaviors have dedicated tests including edge cases (disabled, cached filters, composite keys, non-entity objects).

**Concerns (not blockers):**
- `context.GetService<T>()` resolves from EF's internal service provider, which merges application DI. This works correctly but the comment in `TestTimeProviderOptionsExtension` clarifies this is non-obvious. Consider documenting in `AuditSavePipelineHelper`.
- The `field ??=` lazy logger pattern uses C# 14 field keyword (fine with `LangVersion=latest` on .NET 10).
- No migration scaffold is included (expected — consumers run their own migrations).
- The `extension(IServiceCollection services)` C# 14 syntax is used correctly throughout.
