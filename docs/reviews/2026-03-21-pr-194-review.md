# Code Review Summary

PR #194 introduces valuable messaging resiliency work, but the current branch has four merge-blocking issues:
the public circuit-breaker configuration API does not compile for consumers, half-open recovery is not actually
single-probe and can also stall indefinitely on quiet groups, `RetryProcessor` runtime settings are ignored through
the public DI path, and NATS pause semantics do not stop broker-side intake under an open circuit.

## Reviewers Used

- strict-dotnet-reviewer - correctness, DI wiring, runtime behavior
- pragmatic-dotnet-reviewer - API design, ergonomics, operability
- security-sentinel - failure amplification and abuse resistance
- performance-oracle - backpressure, throughput, resource usage
- code-simplicity-reviewer - dead abstractions and unnecessary complexity
- agent-native-reviewer - agent discoverability of new configuration surface
- learnings-researcher - docs/solutions pattern scan

## Key Findings

- `CircuitBreakerOptions` and `ConsumerCircuitBreakerOptions` use `init` setters behind mutable `Action<TOptions>` APIs, so the advertised configuration examples do not compile.
- Half-open resumes full traffic instead of one probe, and quiet groups can remain stuck in HalfOpen because retries are also blocked there.
- `MessageNeedToRetryProcessor` reads nested `MessagingOptions.RetryProcessor`, but `AddHeadlessMessaging` never copies configured retry-processor values into that DI snapshot.
- NATS uses only a local pause gate while the JetStream push subscription continues delivering messages, which can create blocked callbacks, redeliveries, and resource growth under load.
- `WithCircuitBreaker(...)` currently depends on fluent call order and can bind overrides to stale metadata.
- Group client pause and resume logic iterates a mutable list without synchronization while startup can still be adding clients.
- No relevant known patterns were found in `docs/solutions/`; the existing `docs/solutions/publish.md` doc is unrelated.
