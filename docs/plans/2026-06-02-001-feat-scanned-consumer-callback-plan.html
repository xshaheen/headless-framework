Summary

Add a callback-bearing overload of ForMessagesFromAssembly (and its …Containing<TMarker> sibling) that runs a caller-supplied Action<ScannedConsumerContext, IScannedConsumerBuilder> once per discovered IConsume<TMessage>, so scanned consumers can be routed to the queue lane, given concurrency / group / handler-id / circuit-breaker, or skipped. The existing no-arg overload is refactored to delegate through the same path with a no-op callback, so its Bus / concurrency-1 behaviour is preserved by construction.

Problem Frame

Assembly scanning is the only ergonomic way to register many consumers at once, but it is a one-shape door: MessageRegistrationFactory.CreateScanned hardcodes IntentType.Bus and concurrency 1 for every discovered consumer. A consumer authored for the queue lane that happens to live in a scanned assembly silently registers as a broadcast Bus consumer, and the misconfiguration only surfaces at runtime as "messages aren't load-balancing the way I expected." The underlying registration record already carries intent, group, concurrency, handler-id, and a circuit-breaker override — there is simply no per-consumer seam in the scan path to set them. This plan adds that seam.

Key Technical Decisions

KTD1

Unify the scan path — no-arg overload delegates through the callback path

Both public overloads route to one private helper (_ScanAssembly(assembly, configure?) in src/Headless.Messaging.Core/Setup.cs). The no-arg overload passes a null / no-op callback, which yields untouched Bus / concurrency-1 registrations — so R6 and R12 hold by construction rather than by parallel maintenance of two scan loops. Trade-off: this modifies a currently-working method, accepted for the single-path simplification.

KTD2

Wrap the existing MessageConsumerRegistrationBuilder; mirror ConsumerBuilderBase validation

The internal mutable MessageConsumerRegistrationBuilder (src/Headless.Messaging.Core/Registration/ConsumerBuilders.cs) already carries Group / Concurrency / HandlerId / CircuitBreakerOverride and an isAssemblyScan ctor flag, with a Build() that emits the MessageConsumerRegistration record. The new ScannedConsumerBuilder wraps the same object and reuses the same Argument.* guards, guaranteeing R8 parity without duplicating guard logic.

KTD3

OnBus() / OnQueue() must reset intent on the backing builder

In the explicit path, intent is fixed at construction (new MessageConsumerRegistrationBuilder(type, intentType)); the lane-specific interfaces carry no intent setter. The scanned builder constructs its backing object as Bus and must let OnQueue() (and a redundant OnBus()) re-set the intent field before Build(). This is the one mechanically non-obvious point of the change — surfaced by research, not the issue.

KTD4

Skip() drops the consumer before DI registration

Today's scan loop registers DI descriptors for every discovered consumer before building its registration. The new loop must run the callback first and, when the builder is skipped, register neither DI descriptors nor a MessageRegistration — making "excluded entirely" (R7) literally true. Skip() is terminal; other builder calls on a skipped consumer are moot. (see origin)

KTD5

Callback shapes only the scanned registration; discovery is untouched

The callback feeds the scanned MessageRegistration that enters _DiscoverMessageRegistrations. Precedence (explicit wins), two-pass name resolution, idempotent merge, and dedup are unchanged (R10, R11). A callback flipping OnQueue() on a pair also registered explicitly silently no-ops — consistent with the documented precedence model, not a warning. (see origin)

KTD6

No MessageName on the scanned builder — scanning stays inference-only

Scanning infers message identity; explicit ForMessage<T> asserts it. Omitting MessageName keeps the scanned builder's job to lane + runtime knobs, and sidesteps the cross-type name-collision sweep, since the callback can never introduce a new message name during scanning. (see origin)

KTD7

New public types carry [PublicAPI]; no analyzer baseline to update

The repo has no PublicAPI.Shipped.txt / Roslyn public-API analyzer baseline — public-surface discipline is the [PublicAPI] attribute convention (per CLAUDE.md). ScannedConsumerContext and IScannedConsumerBuilder carry [PublicAPI], mirroring the existing builder interfaces; no snapshot file changes.

High-Level Technical Design

The new per-consumer path. Three public overloads converge on one private _ScanAssembly helper; the only new control-flow branch is per-consumer Skip(), which suppresses both DI registration and the message registration. Everything downstream of the configured registration is the existing discovery pipeline, unchanged.

Execution note

Start with a failing test for Skip()-excludes-from-DI (AE3) — it pins the callback-before-DI ordering (KTD4) that the refactor must preserve.

Approach & patterns
  • Extract _ScanAssembly(Assembly assembly, Action<ScannedConsumerContext, IScannedConsumerBuilder>? configure) from the current ForMessagesFromAssembly body (Setup.cs ~L54–75). Per discovered (consumerType, messageType): construct a ScannedConsumerBuilder, invoke configure?.Invoke(new ScannedConsumerContext(...), builder), then branch on skip.
  • Skip branch (KTD4): when skipped, continue before any Services.TryAdd — no consumerType descriptor, no IConsume<T> descriptor, no MessageRegistration singleton.
  • Non-skip branch: register the same two DI descriptors as today, then AddSingleton the configured MessageRegistration (built from the ScannedConsumerBuilder) instead of MessageRegistrationFactory.CreateScanned(...). Keep CreateScanned only if still used by the no-op path, else fold it in.
  • No-arg ForMessagesFromAssembly(Assembly) → _ScanAssembly(assembly, null). New overloads → _ScanAssembly(assembly, configure). ForMessagesFromAssemblyContaining<TMarker>(configure) → delegate with typeof(TMarker).Assembly.
  • Patterns to follow: existing extension-member shape in Setup.cs; Argument.IsNotNull(assembly) / Argument.IsNotNull(configure) guards on the public overloads; XML doc on each new overload (null-safe ctx.MessageType.Namespace note).
Test scenarios & suite design

Suite: unit, in ForMessageRegistrationTests.cs, mirroring should_scan_assembly_consumers_as_bus_registrations and should_not_scan_explicitly_registered_consumer_into_bus_lane.

  • Covers AE1. Callback OnQueue() → consumer registers on the queue lane (IntentType.Queue).
  • Covers AE2. Mixed assembly in one scan call — some OnQueue(), some untouched Bus — resolve to their respective lanes.
  • Covers AE3. Skip() excludes the consumer from the registry and from DI (not resolvable as IConsume<T> via the scan path).
  • Covers AE4. Untouched consumer == Bus / concurrency 1, identical to the no-arg overload's output for the same assembly.
  • Covers AE5. Explicit ForMessage<T> wins over a configured scan of the same pair (callback OnQueue() does not override).
  • Covers AE6. Callback Concurrency(0) / empty Group throw with the explicit-builder exception types.
  • Idempotent re-scan: scanning the same assembly twice yields an identical registration set (no double-register, no intent flip) — from learnings (messaging-keyed-di-lock-isolation).
  • Per-consumer WithCircuitBreaker(...) on a scanned consumer flows through to the resolved metadata's circuit-breaker override.
  • Two distinct scanned consumers configured to the same (name, group, intent) via HandlerId throw the competing-identity error at startup (R9 fail-fast).

All nine U2 scenarios pass; the pre-existing scan and precedence tests still pass unchanged (R12). make test-project-fast TEST_PROJECT=tests/Headless.Messaging.Core.Tests.Unit green.

Scope Boundaries

Outside this feature's identity

Declarative attribute-based config (couples consumer classes to Headless attributes; the callback subsumes it via ctx.ConsumerType.GetCustomAttribute<…>()) and an assembly-wide blanket Group default (competing-consumer footgun). (see origin)

Deferred — escape hatch exists

MessageName override on the scanned builder (route renames through explicit ForMessage<T>) and predicate / filter overloads beyond the callback (Skip() already supports broad-scan-then-prune). (see origin)

Deferred to follow-up work

After this lands, capture a docs/solutions/ learning via /dev-compound for the scan-vs-explicit precedence + competing-identity semantics — the learnings search found no doc covering the #361/#362/#367 registration line, and these rules recur for every future registration overload.

Risks & Dependencies

  • Dependency — #362 (satisfied). The builder internals (MessageConsumerRegistrationBuilder, the registration records, discovery precedence) ship from PR #362, merged to main (4380dce39). No blocker.
  • Low — circuit-breaker cardinality. Per-consumer config can multiply distinct group/handler identities; the breaker layer caps tracked groups (MaxTrackedGroups, per circuit-breaker-transport-thread-safety-patterns). This feature exposes the same knobs the explicit path already does, so it adds no new breaker internals — flagged for awareness, not mitigation.
  • Low — handler-id & redelivery dedup. A per-consumer HandlerId override participates in redelivery dedup (per terminal-state-overwrite-on-redelivery). The competing-identity guard already fires on collisions (R9), and the U2 collision test exercises it; no behavioral change beyond what the explicit builder already permits.
